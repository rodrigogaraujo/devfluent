## User Scenarios & Testing *(mandatory)*

### User Story 1 - Developer Starts the Bot (Priority: P1)

A Brazilian developer discovers DevFluent and opens the Telegram bot for the first time. They send `/start` and receive a welcome message in English that introduces the service and begins the onboarding flow. The onboarding collects their English self-assessment, tech profile (role and stack), learning goals (interview prep, meetings, etc.), and optionally a target stack and company type for interview preparation. After the profile collection, a written and speaking assessment determines their level. The bot responds within a few seconds at each step, feels responsive, and the developer understands what DevFluent offers.

**Why this priority**: This is the absolute first touchpoint. If the bot does not respond to `/start`, nothing else matters. This story validates that the entire backend stack is wired together: Telegram webhook, server, and a basic AI-generated greeting. The tech profile and goals are collected upfront because they drive content personalization throughout the entire product experience.

**Independent Test**: Can be fully tested by sending `/start` to the bot in Telegram and completing the full onboarding flow (self-declaration, tech profile, goals, written assessment, speaking assessment). Delivers a functional bot that proves end-to-end connectivity and personalization setup.

**Acceptance Scenarios**:

1. **Given** the bot is deployed and running, **When** a new user sends `/start`, **Then** the bot responds with a welcome message within 5 seconds.
2. **Given** the bot is deployed and running, **When** a new user sends `/start`, **Then** a user record is created in the database with their Telegram ID and default level (Level 1 — Foundation).
3. **Given** a user who already sent `/start` before, **When** they send `/start` again, **Then** the bot recognizes them and does not create a duplicate record.
4. **Given** a new user is in the onboarding flow, **When** they reach the tech profile step, **Then** the bot asks their role (single-select) and technologies (multi-select with confirm button), and stores these in their user profile.
5. **Given** a new user is in the onboarding flow, **When** they reach the goal setting step, **Then** the bot asks what they want to practice (multi-select: HR interview, technical interview, meetings, leading). If they select "Technical interview", the bot additionally asks their target stack and target company type.
6. **Given** a user has completed onboarding with goals, **When** the system generates their first study plan, **Then** the plan content is oriented to their goals and target stack, not just their level.

---

### User Story 2 - Developer Has a First Conversation (Priority: P2)

After the welcome message, the developer sends a text message in English. The AI tutor responds conversationally, adapting to a default beginner level (Level 1 — Foundation). The response is friendly, corrects any errors the developer made, and introduces a new vocabulary word. The conversation context is persisted so follow-up messages are coherent.

**Why this priority**: The core product loop is conversation with the AI tutor. This story validates that the AI integration works, prompts are loaded correctly for Level 1, and conversation history is stored. Without this, the product has no value beyond a greeting.

**Independent Test**: Can be tested by sending 2-3 text messages to the bot after `/start` and verifying the responses are contextually coherent, level-appropriate, and contain corrections if errors are present.

**Acceptance Scenarios**:

1. **Given** a user has completed `/start` and onboarding, **When** they send a text message, **Then** the bot responds with a contextual reply generated by the AI tutor using the Level 1 system prompt, adapted to the user's goals and tech stack.
2. **Given** a user sends a message with a grammar error, **When** the AI processes it, **Then** the response includes an inline correction with a brief explanation in Portuguese.
3. **Given** a user sends multiple messages in sequence, **When** the AI responds to each, **Then** the responses reference prior messages in the same conversation (context is maintained).
4. **Given** a user has had previous conversations on earlier days, **When** they start a new conversation, **Then** the AI tutor references relevant information from past sessions (e.g., "Last time you practiced system design...").

---

### User Story 3 - Developer Sends a Voice Message (Priority: P3)

The developer sends a voice message instead of text. The system transcribes the audio using the speech-to-text service, processes the transcription through the AI tutor, generates a text and audio response, and sends both back via Telegram. The developer hears the tutor's voice in natural English.

**Why this priority**: Audio is central to language learning — developers need to practice speaking and hearing. This story validates the full audio pipeline (STT + AI + TTS) end-to-end. It can be deferred because text-only conversations already deliver value for MVP validation.

**Independent Test**: Can be tested by sending a voice message to the bot and verifying that the bot replies with both text and an audio message generated from the AI response.

**Acceptance Scenarios**:

1. **Given** a user sends a voice message, **When** the system processes it, **Then** the audio is transcribed to text and stored alongside the original audio reference.
2. **Given** a voice message has been transcribed, **When** the AI generates a response, **Then** the response is converted to audio and sent back as a voice message.
3. **Given** a user sends a very short voice message (under 1 second), **When** the system attempts transcription, **Then** the system responds gracefully asking the user to try a longer message.

---

### User Story 4 - Tutor Remembers Past Sessions (Priority: P4)

After a conversation ends (30 minutes of inactivity or `/end` command), the system automatically generates a summary capturing: topics discussed, errors made, corrections given, and new vocabulary learned. On the next conversation, the tutor uses this memory to personalize the experience — referencing past mistakes, reinforcing vocabulary, and building on previously discussed topics.

**Why this priority**: Memory is what differentiates DevFluent from a generic chatbot. Without it, every conversation starts from zero. This story validates the memory architecture: summary generation, error pattern tracking, and multi-layer context assembly. It depends on US2 (conversation) being functional first.

**Independent Test**: Can be tested by having two conversations on separate occasions. In the second conversation, verify the tutor references specific details from the first (e.g., a word learned, an error corrected, a topic discussed).

**Acceptance Scenarios**:

1. **Given** a conversation has ended (30-min timeout or `/end`), **When** the system processes the session end, **Then** a summary of 3-4 sentences is generated and stored, capturing topics, errors, corrections, and new vocabulary.
2. **Given** a conversation summary has been generated, **When** the user starts a new conversation, **Then** the AI tutor's context includes relevant summaries from past sessions.
3. **Given** a user has made the same grammar error across multiple conversations, **When** the system detects this pattern, **Then** the error is tracked with an occurrence count and surfaced in the tutor's context as a recurring mistake.
4. **Given** the context assembly process runs, **When** it builds the prompt for the AI, **Then** the total context stays within a defined token budget (not exceeding capacity limits) by selecting the most relevant information.

---

### Edge Cases

- What happens when the Telegram webhook receives a message type the system does not support (e.g., sticker, location, document)? The bot MUST reply with a friendly message explaining it only supports text and voice messages.
- What happens when the AI service is unavailable or times out? The bot MUST reply with a user-friendly error message ("I'm having trouble thinking right now. Try again in a moment!") and log the error.
- What happens when the STT service fails to transcribe audio? The bot MUST reply asking the user to try again or switch to text, and log the transcription failure.
- What happens when a user sends messages faster than the system can process? The system MUST queue messages and process them in order, never dropping messages silently.
- What happens when the database is unreachable? The bot MUST respond with a generic error message and alert the monitoring system.
- What happens when a conversation has no messages (user never responded after `/start`)? The system MUST NOT generate an empty summary.
- What happens when summary generation fails? The conversation MUST still be considered complete; the summary is best-effort and the system logs the failure without affecting the user.
- What happens when the context budget is exceeded due to long history? The system MUST truncate older context (least recent summaries first) to stay within limits.
- What happens when a non-activated user tries to send a message or voice note? The bot MUST reply with a waitlist message ("You're on the waitlist! We'll activate you soon.") and NOT process the message through AI services. The user record is preserved for future activation.

## Clarifications

### Session 2026-02-22

- Q: When a user sends a message after a previous conversation ended (timeout/`/end`), how is a new conversation created? → A: Auto-create silently. Any message sent when no active conversation exists automatically creates a new one. The user never sees "sessions" — it's internal infrastructure. `/start` is only for first-time onboarding. The system checks for an active conversation; if none exists (or previous timed out), a new one is created transparently.
- Q: What happens when a non-activated user sends `/start`? → A: Bot responds to `/start`, creates user record (is_active=FALSE), but gates all tutoring features behind `is_active`. Inactive users see a waitlist message ("You're on the waitlist! We'll activate you soon."). This controls API costs (~R$21/user/mês) while capturing leads. Activation is manual via admin command during the 10-tester phase.
- Q: How does onboarding collect the user's background and learning goals? → A: The onboarding flow has 5 phases: (1) Self-declaration of English level, (2) Tech Profile — role (single-select) + stack (multi-select with confirm), (3) Goal Setting — learning goals (multi-select) + conditional target stack and company type if "Technical interview" is selected, (4) Written Assessment adapted to tech profile, (5) Speaking Assessment adapted to goals. The level determines English complexity; the goals determine content focus. Study plans, mock interviews, and system prompts are all personalized by goals + target_stack.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST receive and process Telegram webhook events (text messages, voice messages, and bot commands).
- **FR-002**: System MUST respond to the `/start` command with a welcome message and create a user record (`is_active=FALSE`) if one does not already exist. All tutoring features (conversation, voice, context assembly) MUST be gated behind the `is_active` flag. Non-activated users MUST receive a friendly waitlist message instead of tutoring responses. Activation is performed manually by an admin during the 10-tester phase.
- **FR-002a**: The onboarding flow MUST collect the user's tech profile (role and stack) and learning goals before the assessment. The flow is: (1) Self-declaration of English level, (2) Tech role (single-select: Backend, Frontend, Fullstack, Mobile, Data/ML, DevOps/Infra), (3) Tech stack (multi-select from common technologies), (4) Learning goals (multi-select: HR/Behavioral interview, Technical interview, Daily meetings, Leading meetings), (5) If "Technical interview" selected: target stack (multi-select) and target company type (single-select: Big Tech, Startup, Enterprise, Not sure), (6) Written assessment, (7) Speaking assessment.
- **FR-002b**: System MUST store the user's tech profile (role, stack) and goals (learning goals, target stack, target company type) in the user record. Users MUST be able to update their goals later via a dedicated command without repeating the full onboarding.
- **FR-003**: System MUST process text messages through the AI tutor and return a contextual response adapted to the user's current level and goals.
- **FR-004**: System MUST transcribe voice messages to text using a speech-to-text service before processing through the AI tutor.
- **FR-005**: System MUST generate audio responses from the AI tutor's text using a text-to-speech service.
- **FR-006**: System MUST persist all conversations and messages in the database with metadata (user level, mode, timestamps). When a user sends a message and no active conversation exists (first message ever, or previous conversation ended via timeout or `/end`), the system MUST silently create a new conversation. The `/start` command is reserved for first-time onboarding only.
- **FR-007**: System MUST assemble context from three layers before each AI interaction: (1) user profile including tech role, stack, goals, target stack, target company, and recurring errors (~300 tokens), (2) current conversation history (last 15 messages), (3) summaries from previous conversations (last 5 most recent).
- **FR-008**: System MUST load and apply level-specific system prompts when generating AI tutor responses (Level 1 as default for new users). Prompts MUST incorporate the user's goals and tech stack to adapt content (e.g., interview terminology for interview-prep users, meeting vocabulary for daily-meetings users).
- **FR-009**: System MUST store audio files (user voice messages and generated TTS audio) in cloud storage and reference them by URL.
- **FR-010**: System MUST validate all incoming webhook payloads before processing.
- **FR-011**: System MUST implement rate limiting to prevent abuse of AI services.
- **FR-012**: System MUST log all errors to the monitoring service with structured context (user ID, conversation ID, component name).
- **FR-013**: System MUST load all configuration from a centralized, typed configuration module — never from environment variables directly in application code.
- **FR-014**: System MUST generate a conversation summary when a conversation ends (30-minute inactivity timeout or explicit `/end` command). The summary MUST capture: topics discussed, errors made with corrections, and new vocabulary learned.
- **FR-015**: System MUST track recurring user errors by type (grammar, pronunciation, vocabulary) with occurrence counts and most recent examples.
- **FR-016**: System MUST keep the assembled context within a token budget to avoid exceeding AI service limits, prioritizing recency for conversation history and relevance for memory summaries.
- **FR-017**: System MUST respond to the `/end` command by ending the current conversation, generating a summary, and confirming to the user that the session has been saved.

### Key Entities

- **User**: Represents a developer using the platform. Identified by Telegram ID. Has a current level (1-4), subscription status, timezone, weekly practice goal, activation status, tech profile (role and stack), and learning goals (goals, target stack, target company type).
- **Conversation**: A session of interaction between a user and the AI tutor. Has a mode (free_chat, mock_interview, assessment, lesson), a topic, and a start/end time. Contains a summary generated by AI at session end, detected errors, and new vocabulary learned.
- **Message**: A single message within a conversation. Can be from the user or the assistant. May contain text, an audio reference, a transcription, corrections, and pronunciation feedback.
- **Study Plan**: A weekly plan generated for a user based on their level. Contains themes, focus skills, and target vocabulary.
- **User Vocabulary**: Individual words a user has learned, with spaced repetition metadata (ease factor, next review date, times seen/used).
- **User Error Patterns**: Aggregated recurring errors per user. Tracks error type (grammar, pronunciation, vocabulary), specific error detail, correction example, occurrence count, and last occurrence date.
- **Assessment**: A periodic evaluation of the user's level. Includes scores across grammar, vocabulary, pronunciation, and fluency.
- **Weekly Metrics**: Aggregated usage statistics per user per week (minutes practiced, messages sent, words learned, streak, XP).

## Assumptions

- The MVP targets 10 internal testers. Infrastructure decisions reflect this scale.
- New users default to Level 1 (Foundation, CEFR A2) until an assessment is completed.
- Conversation context is assembled from three layers: user profile including tech profile and goals (~300 tokens), conversation history (~2-3K tokens), and memory summaries (~500-1K tokens). Total budget: ~4-8K tokens per interaction.
- The Telegram bot uses webhook mode (not polling) for production deployment.
- Audio files are stored in cloud object storage and referenced by URL in the database.
- The system runs as a single modular server with background workers for audio processing and summary generation (no microservices at MVP, per ADR-005).
- Conversation timeout for summary generation is 30 minutes of inactivity.
- Memory summaries use the 5 most recent conversation summaries (simple recency-based retrieval). Semantic/vector-based retrieval is deferred to a future phase.
- Payments are handled manually via PIX for the 10 internal testers. Automated payment integration is deferred.
- The backend lives in a dedicated `backend/` directory within the repository, self-contained with its own dependency management, tests, and migrations. Future web and mobile platforms will be added as sibling directories only after product validation.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A developer can send `/start` and receive a welcome message within 5 seconds of the bot being deployed.
- **SC-002**: A developer can have a 5-message text conversation with the AI tutor where each response is contextually relevant to the previous messages.
- **SC-003**: A developer can send a voice message and receive both a text and audio response within 10 seconds.
- **SC-004**: All user interactions (messages, conversations) are persisted in the database and retrievable.
- **SC-005**: The system handles gracefully all error scenarios (AI timeout, STT failure, unsupported message type) without crashing or leaving the user without a response.
- **SC-006**: After two or more conversations on separate occasions, the AI tutor demonstrates continuity by referencing specific details from past sessions (topics, errors, vocabulary) without the user needing to repeat themselves.
- **SC-007**: Conversation summaries are generated within 30 seconds of a session ending and accurately capture the key topics, errors, and vocabulary from the session.
- **SC-008**: Recurring errors (same mistake type appearing 3+ times across conversations) are detected and surfaced in the tutor's context, leading to targeted correction.
- **SC-009**: A developer who completes onboarding with specific goals (e.g., "Technical interview" with target stack) receives a study plan and mock interview suggestions oriented to those goals, not generic English content.
